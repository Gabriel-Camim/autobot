# DGE 2.0 Metric And Formula Registry

## Purpose

DGE 2.0 must not operate as a black-box calculator.

Every relevant premise, KPI, projection, and derived metric must be traceable to:

- the input values used;
- the formula applied;
- the formula version;
- the calculation engine version;
- the scenario snapshot;
- the timestamp;
- the confidence level;
- the source of the data.

This registry is the official base for finer projections, operational tracking, future integrations, and auditability.

## Core Idea

DGE 2.0 needs two registries:

1. `Metric Catalog`
   - what the DGE knows how to receive, calculate, observe, project, compare, and track.

2. `Formula Registry`
   - how the DGE calculates each derived value, which inputs it depends on, and how the result should be explained.

Together, they make the system expandable without becoming a fragile form with hundreds of loose fields.

## Metric Types

```txt
premise
calculated_kpi
observed_kpi
projection
derived_metric
timeline_event
```

### Premise

A value provided by a user, import, connector, AI extraction, or default assumption.

Examples:

- monthly GMV;
- average ticket;
- marketplace fee;
- delivery cost;
- conversion rate;
- CAC;
- LTV;
- inventory coverage.

### Calculated KPI

A KPI generated by the DGE engine from premises and formulas.

Examples:

- monthly marketplace leakage;
- recovered margin;
- incremental profit;
- payback;
- projected ROI.

### Observed KPI

A real operational KPI imported from systems such as ecommerce, ERP, CRM, logistics, sheets, or manual entry.

Examples:

- actual GMV;
- actual conversion rate;
- actual freight cost;
- actual return rate;
- actual CAC;
- actual inventory turnover.

### Projection

A time-based calculated series.

Examples:

- month-by-month accumulated cash;
- projected orders;
- projected own-channel GMV;
- projected operational capacity;
- projected hub break-even.

### Derived Metric

A secondary value useful for diagnosis, comparison, or explanation.

Examples:

- conversion factor;
- review factor;
- freight friction factor;
- LTV/CAC factor;
- ramp factor;
- growth factor.

### Timeline Event

An event explaining that something changed or was calculated.

Examples:

- scenario calculated;
- premise changed;
- KPI observed;
- formula version changed;
- report generated;
- contract generated;
- integration imported new data.

## Metric Catalog Shape

Each metric should be described before it appears in calculations or UI.

```js
{
  key: 'monthly_marketplace_leakage',
  label: 'Vazamento mensal do marketplace',
  type: 'calculated_kpi',
  domain: 'marketplace',
  unit: 'BRL',
  source: 'calculated',
  cadence: 'snapshot',
  trackable: true,
  requiredFor: ['scenario', 'report', 'contract'],
  formulaKey: 'marketplace.total_leakage',
  formulaVersion: '1.0.0',
  description: 'Custo mensal estimado da dependencia do marketplace incluindo comissao, operacao, perdas comerciais e perda de dados.'
}
```

## Formula Registry Shape

Each formula must be registered as a stable object.

```js
{
  key: 'marketplace.total_leakage',
  version: '1.0.0',
  label: 'Vazamento total do marketplace',
  domain: 'marketplace',
  outputMetricKey: 'monthly_marketplace_leakage',
  inputMetricKeys: [
    'marketplace_gmv',
    'marketplace_fee_percent',
    'marketplace_operating_cost_percent',
    'estimated_price_comparison_loss_percent',
    'cancellation_rate_percent',
    'return_warranty_rate_percent',
    'customers_without_direct_contact_percent'
  ],
  expression: [
    'marketplace_commission_cost',
    '+ marketplace_operating_cost',
    '+ price_comparison_loss',
    '+ cancellation_loss',
    '+ return_loss',
    '+ lost_data_value'
  ].join(' '),
  assumptions: [
    'Cancellation impact uses 45% of cancelled GMV as effective loss.',
    'Return/warranty impact uses 40% of affected GMV as effective loss.',
    'Lost data value uses 1.8% of marketplace GMV without direct customer contact.'
  ],
  confidence: 'modeled',
  owner: 'DGE finance engine',
  status: 'active',
  introducedAt: '2026-05-13'
}
```

## Formula Execution Trace

Every official scenario calculation should emit formula traces.

```js
{
  formulaKey: 'marketplace.total_leakage',
  formulaVersion: '1.0.0',
  outputMetricKey: 'monthly_marketplace_leakage',
  outputValue: 438000,
  unit: 'BRL',
  inputs: [
    { key: 'marketplace_gmv', value: 2000000, unit: 'BRL' },
    { key: 'marketplace_fee_percent', value: 16, unit: 'percent' },
    { key: 'marketplace_operating_cost_percent', value: 5, unit: 'percent' }
  ],
  intermediateValues: [
    { key: 'marketplace_commission_cost', value: 320000, unit: 'BRL' },
    { key: 'marketplace_operating_cost', value: 100000, unit: 'BRL' },
    { key: 'price_comparison_loss', value: 100000, unit: 'BRL' }
  ],
  scenarioSnapshotId: '...',
  calculationRunId: '...',
  calculatedAt: '2026-05-13T00:00:00.000Z'
}
```

## Calculation Run

Each backend calculation should create a calculation run object.

```js
{
  id: 'calc-run-...',
  scenarioSnapshotId: '...',
  engineKey: 'dge-finance-engine',
  engineVersion: '1.0.0',
  formulaRegistryVersion: '1.0.0',
  inputHash: '...',
  status: 'success',
  startedAt: '...',
  finishedAt: '...',
  warnings: [],
  formulaTraces: []
}
```

This is the object that lets DGE answer:

- which engine calculated this result;
- which formulas were active;
- which input values were used;
- whether the result is reproducible;
- whether a future formula update changed the answer.

## Required Formula Metadata

Every formula needs:

- `key`;
- `version`;
- `label`;
- `domain`;
- `outputMetricKey`;
- `inputMetricKeys`;
- `expression`;
- `assumptions`;
- `status`;
- `confidence`;
- `owner`.

Optional but recommended:

- `introducedAt`;
- `deprecatedAt`;
- `replacedBy`;
- `references`;
- `notes`;
- `testCases`.

## Formula Statuses

```txt
draft
active
deprecated
experimental
blocked
```

Only `active` formulas should be used in official calculations unless the request explicitly asks for an experimental scenario.

## Confidence Levels

```txt
exact
modeled
estimated
imported
ai_estimated
manual
unknown
```

Examples:

- `exact`: direct arithmetic from official numbers.
- `modeled`: deterministic model with business assumptions.
- `estimated`: user supplied estimate.
- `imported`: real operational value from a connected source.
- `ai_estimated`: extracted or inferred by AI and awaiting confirmation.

## Persistence Model

Recommended Postgres tables for DGE 2.0:

```txt
metric_catalog
formula_registry
calculation_runs
formula_execution_traces
kpi_snapshots
premise_snapshots
timeline_events
```

The current schema already has `scenario_snapshots`, `scenario_premises`, and `kpi_snapshots`. DGE 2.0 should extend from that base instead of duplicating it.

## API Shape

`POST /api/scenarios/calculate` should eventually return:

```js
{
  scenarioSnapshot,
  calculationRun,
  premises,
  kpis,
  projections,
  projectionStates,
  formulaTraces,
  timelineEvent,
  memory,
  warnings
}
```

Current implementation note:

`projectionStates` is already returned by `POST /api/scenarios/calculate`. It explains every projected month with objectives, projection families, formulas used, ramp and growth drivers, monthly outputs, milestones, and a human-readable explanation.

For UI performance, formula traces can be summarized in normal responses and fetched fully through:

```txt
GET /api/calculation-runs/:id
GET /api/calculation-runs/:id/formula-traces
```

## UX Requirement

Every important number in DGE 2.0 should eventually support an "explain this number" action.

That explanation should show:

- value;
- formula label;
- formula version;
- inputs used;
- assumptions;
- scenario snapshot;
- calculation time;
- whether the value is calculated, observed, imported, estimated, or AI-estimated.

## Migration Plan

### Phase 1: Catalog

Create a static metric and formula catalog in code.

```txt
server/modules/metrics/metricCatalog.js
server/modules/finance/formulaRegistry.js
server/modules/finance/projectionRegistry.js
server/modules/finance/tracePackRegistry.js
```

Initial implementation status:

- `metricCatalog.js` already covers the current scenario form premises plus future operational categories such as CRM, inventory, hubs, marketing, logistics, operations, and observed KPIs.
- `formulaRegistry.js` already covers the main current finance-engine formula groups: marketplace, own channel, profit, logistics, app/CRM, investment, and projection.
- `projectionRegistry.js` declares projection families and objectives such as financial return, channel migration, logistics efficiency, CRM retention, inventory resilience, and marketing efficiency.
- `tracePackRegistry.js` groups formulas, drivers, and outputs into explainable trace packs for each projected month.
- The first live endpoint integration emits traces for the headline KPI formulas returned by `POST /api/scenarios/calculate`.

The catalog is intentionally broader than the current UI. DGE 2.0 should know the full operational vocabulary before every field is displayed or required.

### Phase 2: Trace Current Engine

Wrap the current finance engine so it emits formula traces without changing the final outputs.

Start with headline KPIs:

- marketplace leakage;
- recovered GMV;
- recovered margin;
- incremental profit;
- payback month.

Then expand into intermediate formulas:

- conversion factor;
- review factor;
- freight friction factor;
- LTV/CAC factor;
- logistics savings;
- app retention upside;
- monthly projection rows.

### Phase 3: Persist Runs

Persist calculation run metadata and formula traces through `memoryRepository`.

### Phase 4: UI Explanation

Add "explain number" capability for headline KPIs and projection rows.

### Phase 5: Observed KPIs

Add operational/imported KPIs and compare:

```txt
projected vs observed
```

This is where DGE becomes a living operational intelligence layer, not only a simulator.

## Non-Negotiable Rule

No official DGE 2.0 calculation should exist without a formula key and version.
